# OpenClaw 与猎头场景及本项目 Skills 对照分析

## 说明

- **OpenClaw 背景**：下文对 OpenClaw 的定位与能力边界，依据你提供的行业叙述整理；具体版本、仓库数据与商标沿革以官方/GitHub 为准。
- **本项目 skills**：依据仓库内 `skills/builtin/headhunter_shared/references/skill-inventory.md`、`src/lib/builtin-skills.ts` 及《项目场景适配性分析》《猎头IP打造能力方案》中的结论做静态对照。
- 仓库内**未发现** `cursor_project_rules` 目录，分析以当前项目文件为准。

## 一、对猎头顾问而言：OpenClaw「能做什么」

OpenClaw 类框架的核心是：**在授权与规则下，让 AI 不只回答，而是连接工具与流程去执行任务**。映射到猎头日常，可归纳为三类「可自动化、可编排」的工作：

### 1. 网上找人（检索、监控、初筛）

- 按目标公司、关键词在招聘/职业平台检索线索。
- 定时或按需扫描公开渠道上的候选人动态（发帖、项目、开源、学术等，视平台与合规而定）。
- 对检索结果做第一层匹配与排序，输出待跟进名单或候选池。

### 2. 管理人才库（结构化、标签、合并）

- 将候选人登记进表格或业务系统。
- 从简历/简介中抽取字段，打标签（技术栈、职级、行业、项目规模等）。
- 合并多来源信息、去重、形成可检索的结构化记录。
- 按方向/行业/职级分类存储，支撑后续寻访与推荐。

### 3. 联系与触达（文案、渠道、记录）

- 根据简历与岗位生成个性化或定向开场文案。
- 在规则内提取可用联系方式、生成邮件草稿或消息模板（是否自动发送取决于产品与合规策略）。
- 记录触达行为与结果，支撑后续跟进节奏。

### 4. 与「类 OpenClaw」一致的长期限制（产品层面）

- **部署与配置门槛**仍偏高，需要流程设计与运维意识。
- **权限、安全、稳定性**需人为治理（账号、数据出境、平台 ToS 等）。
- **效果上限**依赖使用者如何把业务拆成可执行步骤，而非模型本身。

## 二、对猎头顾问而言：OpenClaw「长期不能替代什么」（护城河）

下列能力属于**人际信任、判断与责任**，框架可辅助信息处理，但**不能替代人**承担终局：

| 维度 | 含义（猎头场景） |
|------|------------------|
| **信任** | 真实信誉、长期关系、让候选人说真话、为推荐背书。 |
| **动机与情绪** | 口是心非、犹豫/焦虑/抗拒、矛盾心理、未说出口的限制条件。 |
| **决策与取舍** | 是否继续推、何时停止、是否值得消耗关系与信用。 |
| **博弈与影响** | 临场谈判、非理性应对、改变对方认知的时机与方式。 |
| **责任与后果** | 对推荐结果负责、对失败关系兜底、灰度地带的职业判断。 |

**结论**：若顾问的工作可被类 OpenClaw 系统在无信任增量的情况下完整替代，则岗位价值会迅速被压缩；护城河应建在上述维度，同时用 AI 吃掉重复、低判断密度的环节。

## 三、信任与个人 IP：AI 能助力什么

个人 IP 是**信任的可见载体**之一：持续、一致、可验证的专业输出，降低候选人与甲方的「初次信任成本」。

AI 较适合助力：

- **把一线洞察结构化成内容**（岗位分析、市场观察、沟通要点提炼）。
- **多平台改写与节奏建议**（同一素材适配不同渠道）。
- **选题与素材库沉淀**（从业务数据中拉选题，而非纯空想文案）。

AI 较难单独完成：

- **可信度来源**（真实案例、可核对的经历、长期一致的人设）。
- **关系中的承诺与兜底**（候选人最终是否信你，仍取决于线下行为）。

## 四、本项目 Skills 已覆盖哪些（与 OpenClaw 类「能做的事」对照）

以下对应**仓库已注册的内置猎头技能**（见 `src/lib/builtin-skills.ts` 与 `skill-inventory.md`）。

| 能力域 | OpenClaw 叙述中的典型能力 | 本项目对应情况 |
|--------|---------------------------|----------------|
| 岗位/JD 情报 | 看招聘网站、拉岗位列表与 JD | **`headhunter-find-job`**：浏览器自动化获取岗位与 JD。 |
| 岗位深度分析 | 结构化岗位认知、写入系统 | **`headhunter-search-report`**：岗位分析报告，对接飞书文档与岗位库。 |
| 网上找人 / 排序 / TopN | 多源寻访、初筛、排序 | **`headhunter-candidate-sourcing`**：多源寻访、打分、Top10、`handoffs` 分流。 |
| 简历与 JD 匹配 | 深挖单份简历、面试题与风险点 | **`headhunter-cv-jd-matching`**：本地简历与 JD 匹配、可疑点、面试辅助问答等（非全网寻访）。 |
| 首次触达与节奏 | 首触文案、首轮跟进 | **`headhunter-outreach-message`**：渠道建议、首触文案、跟进节奏。 |
| 已建联跟进 | 保温、节点推进 | **`headhunter-greeting-skill`**：热度与下一步动作、微信侧能力（见技能定义）。 |
| 客户侧维护 | 续单、风险、关怀节奏 | **`headhunter-client-nurture`**：客户分层、触点建议（非候选人长期建档的唯一入口）。 |
| 人才库 / 项目工作台 | 表、字段、记录、文档 I/O | **`headhunter-table-manage`**：飞书工作台维度的岗位库/人才库/进展管理。 |
| 正式交付文档 | 推荐报告 | **`headhunter-candidate-report`**：候选人推荐报告写入飞书。 |
| 入口与扩展 | 通用问答、技能发现 | **`headhunter-chat`**、**`skill-creator`**。 |

**归纳**：在「编排 + 工具」这一层，本项目已通过**技能化**覆盖了与 OpenClaw 叙述高度重叠的 **寻访、匹配、触达、报告、飞书工作台** 主线；与「全自动 Agent 平台」的差异主要在于：**产品形态是「技能驱动的猎头助手」**，而非独立开源 Agent 运行时；浏览器自动化、飞书、脚本能力已部分体现「真的做事」。

## 五、本项目仍缺什么（相对 OpenClaw 全谱 + 猎头 IP）

### 1. 与 OpenClaw「理想清单」相比的常见缺口

- **全时定时监控**（如长期监控 LinkedIn/脉脉/GitHub 动态）：当前以**任务驱动脚本与技能**为主，未见完整「订阅式监控产品化」描述。
- **跨平台深度统一**：依赖各平台规则与 Playwright/接口条件，**合规与稳定性**需持续投入。
- **自动发送与全自动触达**：技能侧强调文案与节奏，**是否自动发、如何审发**仍依赖人与产品策略（与你的叙述一致）。

### 2. 与《项目场景适配性分析》一致的缺口

- **公司分析**：部分具备「公司招聘情报」，缺独立公司研究模块。
- **面试辅导**：匹配链路中有面试建议，**缺独立辅导闭环**（纪要 → 知识库 → 多轮模拟等）。
- **薪酬调研**：有岗位与薪资字段、对标模板，**缺独立薪酬数据库与调研引擎**。

### 3. 与《猎头IP打造能力方案》一致的缺口（信任基建）

当前项目**尚未具备完整「猎头 IP 打造」技能或页面**，包括但不限于：

| IP 模块 | 状态（方案文档结论） |
|---------|----------------------|
| 人设定位 | 未作为独立能力落地 |
| 选题引擎 | 未系统化接入业务数据 |
| 内容生成与多平台改写 | 未形成独立技能/流水线 |
| 内容资产库 | 无 |
| 发布节奏与分发 | 无 |
| 转化承接（私信/线索/跟进） | 未与内容侧打通 |

**已有可复用底座（与 IP 素材相关）**：岗位分析、简历匹配与风险点、沟通/触达类技能，可作为 IP 的**素材来源**，但**不等于 IP 运营系统**。

## 六、一句话对齐

- **OpenClaw 类能力**：适合吃掉猎头流程中**可编排、可记录、可重复**的环节；**信任、判断、责任**仍属人。
- **本项目**：在**寻访—匹配—触达—报告—飞书工作台**上已有成体系的 **skills**；**IP 化信任资产**与**完整监控/公司研究/面试辅导闭环/薪酬引擎**仍是主要增量空间。

若你希望下一步落地 IP，建议优先按《猎头IP打造能力方案》中的 **`headhunter-ip-builder` 类技能 → 资产库 → 发布与转化`** 顺序演进，并与现有 `headhunter-search-report`、`headhunter-cv-jd-matching`、`headhunter-outreach-message` 等产出显式对接。
